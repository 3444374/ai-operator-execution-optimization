# opening/AGENTS.md

本目录维护开题相关材料，包括开题报告、开题汇报 PPT、飞书进度汇报、文献精读、答辩问答和材料同步日志。进入本目录前先读根目录 `AGENTS.md`，再读本文件和 `README.md`。

## 工作定位

- 开题不是证明课题已经完成，而是证明题目有研究价值、问题定义清楚、技术路线可执行、前期准备有基础、后续计划能落地。
- 开题材料不能写成项目宣传稿。必须主动说明当前证据、边界、风险和后续验证计划。
- 当前汇报主线优先采用：先讲数据库 AI 算子带来的新执行链路问题，再讲已有实验发现的链路瓶颈，最后讲后续优化方向。

## 目录职责

- `report/`：开题报告正文、Word 转写材料、学校模板适配记录。
- `slides/`：开题 PPT 源稿、讲稿备注、PPTX 输出和模板适配说明。
- `feishu/`：飞书进度汇报稿、周/月度同步材料。
- `literature/`：参考文献清单、精读文献笔记、CCF-A 优先候选。
- `assets/`：PPT 可复用图、SVG、表格、流程图和模板素材说明。
- `logs/`：非实验类 project log，记录开题材料、文献、PPT、飞书同步变更。

## 写作规则

- 结论必须有来源和边界：文献、官方文档、本地实验事实、合理推断、待验证问题要分开写。
- 不把 Ray 写成唯一研究对象。更稳的表达是“数据库 AI 算子的外部执行链路优化”或“模型服务感知外部执行链路优化”。
- 不把开题阶段的初步实验写成最终贡献。使用“可行性依据”“初步信号”“后续将扩大验证”等表述。
- 创新点写成预期创新点，并与研究内容、技术路线、评价指标一一对应。
- 正文和 PPT 避免 AI 味：少用空泛意义、宣传式表述、过度强调、长段落和没有依据的“重要意义”。

## PPT 规则

- 学校模板是版式约束，不是背景截图。优先复用母版、占位符、主题字体、页眉页脚。
- 正文、图表和流程图必须放在安全内容区，不压页眉、页脚、校徽。
- 一页只突出一个主结论。图中只放关键词，解释放到备注。
- 每页备注默认包含两块：`汇报讲稿` 和 `答辩备注`。
- PPT 生成或修改后必须用 WPS/PowerPoint 实际打开检查。

## 同步规则

- 改题目时，同步报告、PPT、封面、飞书标题和 README。
- 改研究内容时，同步技术路线、进度安排、创新点和答辩备注。
- 改文献时，同步报告参考文献、精读清单、PPT 参考页和文献日志。
- 改实验口径时，同步 `motivation/results/`、`learning/experiment_walkthrough.md`、报告、PPT 和飞书版相关表述。
- 非实验类修改记录到 `opening/logs/project_log.md`，不要写进 experiment log。

## 文献规则

- 文献优先从 CCF-A 会议/期刊和权威系统论文中选择，再补官方文档和工程系统资料。
- 参考文献不能只满足数量。核心精读文献要能回答：解决什么问题、核心方法是什么、和本课题有什么关系、有什么不足。
- Snowflake、pgai、PostgresML、pgvector、Ray、Daft、Lance 等资料要区分论文、官方文档、源码 README 和本地实验事实。
## 飞书同步规则

- 本地 Markdown 是源稿，飞书文档是发布面。
- 开题报告与开题汇报同步到用户给出的开题飞书 wiki。
- 动机测试与可行性测试同步到用户给出的实验飞书 wiki。
- 实际写入飞书前必须先检查本地报告、PPT、实验结果和飞书版口径一致。
- 普通飞书 wiki / docx 使用 `lark-doc`；嵌入 Base 时再用 `lark-base`；直接编辑飞书幻灯片时再用 `lark-slides`。
- 飞书上的实验内容记录标准与项目实验报告保持一致，讲解方式参考 `learning/experiment_walkthrough.md`：必须解释实验问题、链路位置、变量含义、原始结果、结果怎么读、能说明什么、不能说明什么和下一步。
