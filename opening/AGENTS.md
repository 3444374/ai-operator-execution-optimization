# opening/AGENTS.md

本目录维护开题相关材料，包括开题报告、开题汇报 PPT、飞书进度汇报、文献精读、答辩问答和材料同步日志。进入本目录前先读根目录 `AGENTS.md`，再读本文件、`README.md` 和 `navigation.md`。

## 工作定位

- 开题不是证明课题已经完成，而是证明题目有研究价值、问题定义清楚、技术路线可执行、前期准备有基础、后续计划能落地。
- 开题材料不能写成项目宣传稿。必须主动说明当前证据、边界、风险和后续验证计划。
- 当前汇报主线优先采用：先讲数据库成为 AI workload 入口后带来的数据执行链路问题，再讲已有实验发现的 batch、调度、模型服务和写回瓶颈，最后讲 Daft/Arrow、Ray、GPU 服务和 Lance / 数据库 sink 的协同优化方向。

## 目录职责

- `report/`：开题报告正文、Word 转写材料、学校模板适配记录。
- `slides/`：开题 PPT 源稿、讲稿备注、PPTX 输出和模板适配说明。
- `feishu/`：飞书进度汇报稿、周/月度同步材料。
- `literature/`：参考文献清单、精读文献笔记、CCF-A 优先候选。
- `assets/`：PPT 可复用图、SVG、表格、流程图和模板素材说明。
- `logs/`：非实验类 project log，记录开题材料、文献、PPT、飞书同步变更。

## 写作规则

- 结论必须有来源和边界：文献、官方文档、本地实验事实、合理推断、待验证问题要分开写。
- 不把 Ray 写成唯一研究对象。更稳的表达是“数据库驱动 AI workload 的分布式数据执行与存储协同优化”；Daft/Ray/Lance 是系统机制和验证平台，不是简单产品集成题目。
- 不把开题阶段的初步实验写成最终贡献。使用“可行性依据”“初步信号”“后续将扩大验证”等表述。
- 创新点写成预期创新点，并与研究内容、技术路线、评价指标一一对应。
- 正文和 PPT 避免 AI 味：少用空泛意义、宣传式表述、过度强调、长段落和没有依据的“重要意义”。

## PPT 规则

- 学校模板是版式约束，不是背景截图。优先复用母版、占位符、主题字体、页眉页脚。
- 正文、图表和流程图必须放在安全内容区，不压页眉、页脚、校徽。
- 一页只突出一个主结论。图中只放关键词，解释放到备注。
- 每页备注默认包含两块：`汇报讲稿` 和 `答辩备注`。
- PPT 生成或修改后必须用 WPS/PowerPoint 实际打开检查。

## 同步规则

- 改题目时，同步报告、PPT、封面、飞书标题和 README。
- 改研究内容时，同步技术路线、进度安排、创新点和答辩备注。
- 改文献时，同步报告参考文献、精读清单、PPT 参考页和文献日志。
- 改实验口径时，同步 `motivation/results/`、`learning/experiment_walkthrough.md`、报告、PPT 和飞书版相关表述。
- 非实验类修改记录到 `opening/logs/project_log.md`，不要写进 experiment log。
- 需要查找材料时先读 `navigation.md`：项目总方向查根 `README.md` / `PROJECT_INDEX.md`，实验结果查 `motivation/results/` 或 `feasibility/results/`，学习讲解查 `learning/`，文献证据查 `research/`。
- 报告、PPT、飞书版必须互相同步：报告负责完整论证，PPT 负责现场讲解，飞书负责发布同步，不允许三者形成不同口径。

## 文献规则

- 文献优先从 CCF-A 会议/期刊和权威系统论文中选择，再补官方文档和工程系统资料。
- 参考文献不能只满足数量。核心精读文献要能回答：解决什么问题、核心方法是什么、和本课题有什么关系、有什么不足。
- Snowflake、pgai、PostgresML、pgvector、Ray、Daft、Lance 等资料要区分论文、官方文档、源码 README 和本地实验事实。
## 飞书同步规则

- 本地 Markdown 是源稿，飞书文档是发布面。
- 开题报告与开题汇报同步到用户给出的开题飞书 wiki。
- 动机测试与可行性测试同步到用户给出的实验飞书 wiki。
- 实际写入飞书前必须先检查本地报告、PPT、实验结果和飞书版口径一致。
- 普通飞书 wiki / docx 使用 `lark-doc`；嵌入 Base 时再用 `lark-base`；直接编辑飞书幻灯片时再用 `lark-slides`。
- 飞书上的实验内容记录标准与项目实验报告保持一致，讲解方式参考 `learning/experiment_walkthrough.md`：必须解释实验问题、链路位置、变量含义、原始结果、结果怎么读、能说明什么、不能说明什么和下一步。
## 默认参考 skill

开题工作可按任务需要使用以下 skill。它们是方法参考，不是固定流程；真正执行时必须结合本项目的真实阶段、已有实验结果、导师要求、学校模板和当前材料目标灵活选择。不要无脑同时调用，按当前任务选择最小必要集合。

- `karpathy-guidelines`：用于控制假设、边界、实验结论、代码和文档修改范围。
- `humanizer`：用于开题报告、PPT 讲稿、飞书材料的自然表达，避免 AI 味、宣传式表述和空泛意义。
- `academic-research-suite`：用于开题任务组织、研究计划、实验设计、材料完整性检查和阶段性 review。
- `deep-research`：用于需要系统综述、研究问题收敛、研究空间判断和证据地图时。
- `nature-academic-search`：用于开题文献调研、权威论文检索、引用核验、CCF-A 优先文献筛选和参考文献管理。
- `vibe-research-workflow`：用于 AI 辅助研究的流程控制，保证研究判断、核心结论和学术责任由用户掌握。
- `ppt-master`：用于开题 PPT 的源稿到 PPTX 工作流、模板适配、页面生成、备注和导出检查。
- `nature-paper2ppt`：用于借鉴学术汇报叙事、证据驱动的页面组织、图表选择和自检规则。
- `figure-designer`：用于设计开题核心图，包括问题动机图、技术路线图和实验结果图。
- `nature-figure`：用于需要论文级实验图、组图、导出规范和图表质量审查时。
- `lark-doc`：用于普通飞书 wiki / docx 的读取与同步。
- `lark-base`：用于飞书文档中嵌入 Base / 多维表格时的结构化数据维护。
- `lark-slides`：用于后续直接创建或编辑飞书幻灯片时。

文献调研默认优先使用 `nature-academic-search`；需要较完整的研究版图或综述时，再结合 `deep-research` 和 `academic-research-suite`。如果本地已有项目证据已经足够支撑某个判断，优先使用本项目真实材料，再用 skill 补文献、结构和表达。
