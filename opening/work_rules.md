# 开题工作规则

## 工作目标

开题阶段要证明：

- 题目有研究价值。
- 问题定义清楚。
- 技术路线可执行。
- 前期准备有基础。
- 后续计划能落地。
- 风险和边界自己知道。

开题不是最终成果发布。更稳的说法是“当前实验作为可行性依据，后续会扩大验证”，不要说“已经完整证明”。

开题也不是项目之外的附属材料。报告中确定的题目、研究内容、评价指标和边界，会反向约束后续项目规划、实验优先级和对外沟通口径。修改这些内容时，需要同步检查项目级 `README.md`、`PROJECT_INDEX.md`、`overview/current_direction_and_plan.md`，以及 `opening/README.md`、PPT、飞书源稿和问答材料。

后续每次调整开题报告，需要先判断调整类型：

- 只改语言、格式、图表位置：通常只同步报告、PPT、飞书源稿和日志。
- 改题目、研究问题、研究内容、创新点、技术路线、实验优先级或适用边界：必须同步检查项目级 README、`PROJECT_INDEX.md`、`overview/current_direction_and_plan.md`、`motivation/plans/workloads.md`、`motivation/plans/integration.md`、PPT、飞书源稿、问答材料和项目日志。
- 改实验结论或新增实验依据：必须同步检查对应的 `motivation/results/` 或 `feasibility/results/` 报告、学习材料和开题可行性分析。

## 当前工作顺序

当前阶段先不生成 DOCX，优先级如下：

1. 本地 Markdown 源稿：报告、飞书稿、PPT 源稿先在本地统一口径。
2. 飞书文档补全：飞书文档是发布面，内容来自本地 Markdown。
3. PPT 制作：基于报告和图表做现场汇报版本。
4. PPT 同步飞书：把 PPT 核心内容同步到飞书。
5. DOCX 生成：最后使用学校 Word 模板生成正式提交版。

最终 DOCX 必须使用学校模板，不从空白 Word 文档生成；需要继承模板中的章节字体、正文宋体 / Times New Roman、小四号、1.5 倍行距、两侧对齐、图表标注和参考文献格式。

## 写作原则

- 少用空泛判断。不要只写“具有重要意义”，要写清楚重要在哪里、对应什么问题、后续怎么验证。
- 研究内容不能写成工程任务清单。每一项研究内容要按“问题、技术难点、拟采用方法、评估方式”组织，回答为什么难、怎么解决、如何证明有效。
- parser、worker、endpoint、写回、实验脚本等工程步骤放在技术路线或进度安排里；研究内容必须写成可检验的研究问题。
- 研究内容需要包含对比对象、评价指标、消融实验和适用边界，避免写成“把几个模块做完”。
- 创新点要克制。开题阶段写“预期创新点”，不要写成已经完成的创新成果。
- 所有结论都要有边界：当前能说明什么、不能说明什么、后续怎么验证。
- 写实验结论和分析实验数据时，参考 `learning/AGENTS.md` 的精细程度：说明实验目的、链路流程、参数含义、数据来源、结果读法、不能证明什么、下一步怎么验证，并区分本地实验事实、模拟实验事实、合理推断、待确认问题和不能声称的结论。
- 报告、PPT、飞书版、Word 版不能出现不同题目、不同研究内容或不同实验口径。

## 高频答辩策略

老师质疑时，回答结构优先采用：

```text
先承认问题存在
  -> 说明当前为什么这样处理
  -> 说明后续如何补充验证
```

不要和老师争“已经证明了”。开题阶段主动承认边界通常更稳。

## 最终检查

- 题目、研究目标、研究内容一致。
- 报告结构符合学校模板。
- DOCX 使用学校 Word 模板生成，并继承模板格式。
- 参考文献数量、外文比例和精读文献数量满足要求。
- 15 篇精读文献能讲清楚。
- PPT 和报告口径一致。
- 技术路线能对应后续实验和实现任务。
- 创新点没有夸大。
- 局限和风险已主动写入。
- 答辩备注准备了常见问题。
- 所有非实验类修改已记录到 `logs/project_log.md`。
